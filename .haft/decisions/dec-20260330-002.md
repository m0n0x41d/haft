# Drop + consistency cleanup

## 1. Problem Frame

**Signal:** Refresh is clean against the active haft store, but the repo still contains a 266-file, 3.4MB legacy .quint tree including old projections and a local .quint/quint.db. Current code uses .haft and ~/.haft, while some docs and migration code still mention .quint for backward compatibility.

**Constraints:**
- Deleting .quint is destructive; user explicitly asked to drop it if safe.
- Must verify no current runtime path requires .quint to exist for normal operation.
- Should distinguish code/docs that merely mention .quint for migration/history from anything that actively depends on it.
- Do not silently remove backward-compatibility code unless the user asks.

**Acceptance:** Establish whether deleting .quint is safe for current operation; if yes and you confirm, remove .quint and report any remaining references that are migration/docs-only or need follow-up.

## 2. Decision

**Selected:** Drop + consistency cleanup

The active system already runs on .haft and ~/.haft, so keeping the in-repo .quint tree only preserves confusion. We should remove the legacy projection tree and fix obviously stale repository-local references, while keeping migration code that still imports old layouts for older environments.

**Invariants:**
- Normal runtime and init flow must continue to work without .quint present in the repo.
- Migration code may still reference .quint/quint.db as an import source for older projects.
- Only obviously stale repo-local references are updated; backward-compatibility code paths remain intact.

**Post-conditions:**
- [ ] Legacy .quint directory is removed from the working tree.
- [ ] Repo-local config/doc references that imply .quint is the current active path are updated or removed.
- [ ] Refresh and basic project diagnostics still succeed after removal.

**Admissibility:**
- NOT: Do not remove migration code that reads old .quint state.
- NOT: Do not keep repo-local config pointing at deleted .quint binaries.
- NOT: Do not broaden the cleanup into unrelated documentation rewrites.

## 3. Rationale

**Weakest link:** Reference cleanup can miss stray .quint mentions, leaving the repo mostly-correct but still partly haunted.

## 4. Consequences

**Affected files:** CLAUDE.md, README.md, TODO.md, quint-mcp.json, .quint
