# Line-buffered streaming + native image attachments

## 1. Problem Frame

**Signal:** The terminal UI still glitches assistant text during streaming and only self-heals when the final message arrives. Also, multimodal-capable models advertise image support, but the chat UI cannot attach images via Ctrl+V.

**Constraints:**
- Do not regress existing text/file attachment behavior.
- Keep Bubble Tea/Lip Gloss architecture unless a localized change is clearly better.
- Do not invent broad TUI redesign scope beyond streaming stability and image paste UX unless evidence shows a tight dependency.
- Need to respect provider-specific multimodal request formats instead of fake inline tags for images.

**Acceptance:** 1) Streaming assistant text no longer shows transient word corruption/jumping under normal token streaming. 2) Final render matches streamed render except for incomplete trailing content. 3) Ctrl+V can attach image clipboard content when the selected model supports images. 4) Attached images are submitted as proper multimodal parts, not textified pseudo-tags. 5) Existing file attachment and large-text paste flows still work.

## 2. Decision

**Selected:** Line-buffered streaming + native image attachments

Verified crush now provides two concrete patterns we can borrow without broad redesign: native clipboard image attachments with MIME-aware chips, and a more stable message rendering strategy based on stable items, controlled re-rendering, and avoiding unnecessary wrap churn. We'll implement the minimal subset needed locally: stabilize streamed assistant rendering by reducing partial-message reflow and add Ctrl+V image attachments as proper multimodal parts.

**Invariants:**
- Existing text input and file attachment flows continue to work.
- Image attachments are only sent as multimodal parts when the selected provider/model supports images.
- Final assistant message remains authoritative over streaming preview.
- No broad TUI redesign or unrelated visual refactor.

**Post-conditions:**
- [ ] Introduce typed image attachment support in the TUI/model boundary.
- [ ] Reduce streaming render churn enough to eliminate the known visible glitch class.
- [ ] Run relevant tests and diagnostics, then record measured results.

**Admissibility:**
- NOT: Do not fake image support by embedding binary data into text pseudo-tags.
- NOT: Do not regress non-image paste behavior.
- NOT: Do not replace the whole chat list architecture for this fix.

## 3. Rationale

**Weakest link:** Clipboard image support is platform-sensitive and provider payload construction differs between OpenAI and Anthropic, so edge compatibility needs verification.

## 4. Consequences

**Affected files:** internal/tui/model.go, internal/tui/render.go, internal/tui/chatitem.go, internal/provider/openai.go, internal/provider/anthropic.go, internal/agent/types.go

## Impact Measurement (2026-03-30)

**Verdict:** partial

**Findings:**
Implemented a whitespace-preserving live wrap path for streaming assistant text and gated live assistant rendering through a dedicated streaming renderer. Added typed agent.ImagePart, changed TUI submission to pass typed user parts instead of plain text, and taught OpenAI/Anthropic providers to emit native image content blocks. Added platform-gated native clipboard image reads using go-nativeclipboard and Ctrl+V attachment chips. Targeted tests passed: internal/tui selected tests, internal/provider selected tests, internal/agent, internal/agentloop. Full-package go test on tui/provider together initially hung because some broader tests/packages are slower or interactive; targeted verification passed cleanly. Drift/stale artifacts were already reported by overseer before this work and remain a separate repo-health issue.

**Criteria met:**
- [x] Streaming text no longer uses whitespace-collapsing strings.Fields wrapping during live rendering.
- [x] Ctrl+V can attach clipboard image bytes on supported platforms via native clipboard integration.
- [x] OpenAI and Anthropic user-message conversion now serialize image attachments as native multimodal content parts.
- [x] Existing text/file attachment flow remains intact for text attachments.

**Criteria NOT met:**
- [ ] I did not run a full end-to-end interactive TUI session to observe the streaming glitch visually under real provider traffic.
- [ ] Clipboard image support currently covers native-image clipboard bytes only; crush-style text-path fallback for image file paths is not implemented yet.
