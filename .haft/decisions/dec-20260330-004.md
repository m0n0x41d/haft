# Inline attachment pills above user text

## 1. Problem Frame

**Signal:** After Ctrl+V attaches an image and the message is sent, the TUI shows no visible indication in chat history that an image was attached. Claude Code-style UX shows attachment pills/placeholders in the sent message block.

**Constraints:**
- Do not change provider payload semantics.
- Do not regress existing text/file attachment display.
- Keep the fix localized to TUI/view-model rendering.

**Acceptance:** 1) Sent user messages visibly indicate attached images in chat history. 2) Text/file attachments remain visible/understandable. 3) No change to provider request behavior. 4) Existing tests still pass or are updated to cover the new UX.

## 2. Decision

**Selected:** Inline attachment pills above user text

This is the smallest render-only fix that matches the observed Claude Code behavior closely enough: when a user sends a message with images, the sent message itself should visibly carry attachment pills. The message model already has typed image parts, so we can render them without changing providers or persistence semantics.

**Invariants:**
- Provider payload behavior remains unchanged.
- Input attachment chips remain unchanged.
- Sent user messages with attachments remain readable in narrow terminals.

**Post-conditions:**
- [ ] User-message view data includes attachment metadata derived from typed parts.
- [ ] User-message rendering shows attachment pills above body text.

**Admissibility:**
- NOT: Do not add previews or file-opening behavior in this change.
- NOT: Do not rework chat list structure just to show attachment markers.

## 3. Rationale

**Weakest link:** Saved-message hydration and TUI view-model construction must surface attachment metadata correctly, or the pills will only appear for freshly sent messages.

## 4. Consequences

**Affected files:** internal/tui/render.go, internal/tui/model.go, internal/tui/chatitem.go, internal/agent/types.go

## Impact Measurement (2026-03-30)

**Verdict:** accepted

**Findings:**
User-message view state now carries attachment metadata at send time, and renderUserMessage displays image/file pills above the message body. Input chips and sent-message chips now use the same visual vocabulary, which closes the UX gap shown in the screenshots. Verified with targeted TUI tests covering rendered user-message pills and submit-path attachment propagation.

**Criteria met:**
- [x] Sent user messages with image attachments render visible attachment pills in chat history.
- [x] Messages without attachments continue rendering as full-width user blocks.
- [x] The change is render-only; provider behavior was left untouched.
