# Remove legacy .quint projection tree after haft migration

## Signal

Refresh is clean against the active haft store, but the repo still contains a 266-file, 3.4MB legacy .quint tree including old projections and a local .quint/quint.db. Current code uses .haft and ~/.haft, while some docs and migration code still mention .quint for backward compatibility.

## Constraints

- Deleting .quint is destructive; user explicitly asked to drop it if safe.
- Must verify no current runtime path requires .quint to exist for normal operation.
- Should distinguish code/docs that merely mention .quint for migration/history from anything that actively depends on it.
- Do not silently remove backward-compatibility code unless the user asks.

## Optimization Targets

- Remove confusing legacy state
- Avoid breaking current runtime/setup
- Minimize unnecessary cleanup beyond .quint removal

## Observation Indicators

- Whether active code reads .quint during normal serve/init flows versus migration-only
- Whether any committed config points to binaries or assets under .quint
- Whether docs become misleading after removal

## Acceptance

Establish whether deleting .quint is safe for current operation; if yes and you confirm, remove .quint and report any remaining references that are migration/docs-only or need follow-up.

## Blast Radius

Repository history/projections and possibly docs/config references; runtime behavior only if some code or setup still relies on in-repo .quint existing.

## Reversibility

Low once deleted from working tree, though recoverable from git before commit.
