# Stabilize TUI streaming rendering and add image paste attachments

## Signal

The terminal UI still glitches assistant text during streaming and only self-heals when the final message arrives. Also, multimodal-capable models advertise image support, but the chat UI cannot attach images via Ctrl+V.

## Constraints

- Do not regress existing text/file attachment behavior.
- Keep Bubble Tea/Lip Gloss architecture unless a localized change is clearly better.
- Do not invent broad TUI redesign scope beyond streaming stability and image paste UX unless evidence shows a tight dependency.
- Need to respect provider-specific multimodal request formats instead of fake inline tags for images.

## Optimization Targets

- Streaming text stability
- Image paste usability
- Minimal architectural churn

## Observation Indicators

- Current render path reparses and rewraps the full partial assistant message on every delta.
- Whitespace-collapsing wrap logic likely causes transient word jumps.
- Model registry exposes SupportsImages, but no end-to-end image attachment flow exists.

## Acceptance

1) Streaming assistant text no longer shows transient word corruption/jumping under normal token streaming. 2) Final render matches streamed render except for incomplete trailing content. 3) Ctrl+V can attach image clipboard content when the selected model supports images. 4) Attached images are submitted as proper multimodal parts, not textified pseudo-tags. 5) Existing file attachment and large-text paste flows still work.

## Blast Radius

internal/tui streaming/render pipeline, attachment handling in input submission, provider message-part conversion for multimodal requests, possibly terminal paste handling.

## Reversibility

Moderate: isolated to TUI/provider boundaries if we avoid broad renderer replacement.
