# Require baseline before measure in agent workflow

## Signal

The agent records haft_decision(measure) without first creating a baseline because the local tool schema hides baseline, prompts omit it, and measure only warns after the fact. This produces weak CL1 evidence and misleading UX.

## Constraints

- Do not change artifact-layer fallback semantics unless necessary; manual/CLI users may still need weak CL1 measurement.
- Expose baseline in the same local runtime path the agent uses.
- Prefer blocking agent self-measure without baseline rather than silently downgrading evidence.

## Optimization Targets

- Prevent forgetting baselines
- Align runtime with FPF evidence discipline
- Minimal behavioral churn outside agent workflow

## Observation Indicators

- artifact.Baseline exists and server/MCP exposes it, but internal/tools/haft.go omits it.
- CanMeasure only checks cycle order, not artifact verification preconditions.
- Prompt and SKILL text say verify→measure, not baseline→verify→measure.

## Acceptance

1) Local haft_decision tool supports action="baseline". 2) Agent-side measure path refuses to measure decisions with affected_files but no baseline, with actionable guidance. 3) Prompts/docs teach implement→baseline→verify→measure. 4) Relevant tests pass.

## Blast Radius

internal/tools/haft.go local tool schema/execution, agent prompt/skill docs, possibly present formatting for measure warnings.

## Reversibility

High: tool-layer and prompt changes; artifact-layer warning fallback can remain.
