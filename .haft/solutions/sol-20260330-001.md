# Solutions for: Disposition stale legacy quint-era artifacts after haft migration

Problem: prob-20260330-001

## Variants (2)

### V1. Bulk deprecate legacy quint-era artifacts

Treat all currently stale/drifted quint/plugin-mcp-era decisions as obsolete because their referenced files no longer exist in the current codebase. Clean the queue quickly by deprecating them rather than preserving detailed lineage.

**Strengths:**
- Fastest cleanup
- Low implementation complexity
- Stops false-positive refresh noise

**Weakest link:** Can lose decision lineage for concepts that may still matter after migration/rename.

**Risks:**
- Historical reasoning becomes harder to trace
- Could hide a still-relevant unresolved concern under an obsolete label

### V2. Per-artifact lineage-preserving disposition

Review each stale/drifted artifact individually and classify it as deprecate, supersede, reopen, or waive based on whether its concern still exists in the haft-era codebase. Preserve lineage where the concept survives but the file paths or architecture changed.

**Strengths:**
- Best artifact hygiene
- Preserves useful history
- Avoids over-deprecating still-relevant concerns

**Weakest link:** More artifact churn and manual judgment; classification quality depends on evidence quality per artifact.

**Risks:**
- Longer review path
- Some decisions may still lack enough evidence for a crisp successor mapping
