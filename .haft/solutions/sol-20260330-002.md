# Solutions for: Remove legacy .quint projection tree after haft migration

Problem: prob-20260330-002

## Variants (2)

### V1. Minimal safe drop

Delete the legacy .quint directory only. Keep migration code and existing docs/config references untouched, relying on current runtime already using .haft and ~/.haft.

**Strengths:**
- Smallest codebase change
- Low risk to runtime behavior
- Fastest cleanup of the stale tree

**Weakest link:** Leaves misleading docs and repo-local config pointing at .quint, so the repository still advertises a path that no longer exists.

**Risks:**
- Future contributors may recreate or depend on .quint by mistake
- Broken repo-local config may remain unnoticed until used

### V2. Drop + consistency cleanup

Delete the legacy .quint directory and update obviously stale repo-local docs/config that still claim .quint is the active path. Keep migration/backward-compat code in place so existing external environments can still migrate.

**Strengths:**
- Removes the confusing legacy tree and the most misleading references
- Keeps runtime compatibility code intact
- Leaves the repo aligned with how the tool actually works today

**Weakest link:** Broader diff; easy to miss a stale reference and think the cleanup is complete when it is only mostly complete.

**Risks:**
- Some historical documentation context may be softened or lost
- A non-obvious local workflow could still reference .quint outside the checked files
