# Solutions for: Stabilize TUI streaming rendering and add image paste attachments

Problem: prob-20260330-003

## Variants (4)

### V1. Line-buffered streaming + native image attachments

Keep the current TUI structure, but split assistant streaming into committed complete lines plus one transient tail line. Render committed lines with markdown formatting only after newline boundaries, render the tail as plain text, and finalize on stream completion. Add clipboard image attachment support in the input layer and send images as proper provider multimodal parts instead of inline pseudo-tags.

**Strengths:**
- Directly targets the likely root cause of streaming glitches.
- Localized change: mostly internal/tui render/model plus provider request builders.
- Preserves current chat list and overall UI design.
- Matches proven patterns from Codex/Crush-style TUIs.

**Weakest link:** If glitching is also caused by lower-level viewport/double-wrap behavior, line buffering alone may not fully eliminate visual jumps.

**Risks:**
- Requires introducing a second attachment representation for images instead of current text-only pseudo-tag flow.
- Needs provider-specific request changes for OpenAI and Anthropic multimodal formats.

### V2. Incremental row cache renderer + attachment unification

Replace full-message re-rendering on each delta with a stable incremental row cache keyed by message ID. Streaming deltas append fragments into a row builder that preserves wrapping and row identity across updates. At the same time, refactor attachments into typed parts (text file, binary file, image) at the model boundary and have providers consume those parts directly.

**Strengths:**
- Best technical foundation for stable streaming and future multimodal work.
- Eliminates repeated full-text wrapping and markdown reparsing.
- Creates one canonical attachment model instead of mixing inline text tags and chips.

**Weakest link:** Blast radius is substantially larger than the bug warrants; easy to overengineer.

**Risks:**
- Larger refactor across TUI, agent part modeling, and providers.
- Higher chance of introducing regressions in rendering, scrolling, or tool/result display.

### V3. Minimal surgical fixes: whitespace-preserving wrap + image paste as file attachments

Keep the current architecture and make the smallest set of changes: replace strings.Fields-based wrapping with whitespace-preserving tokenization, stop double wrapping where possible, and add image detection in paste/attachment handling by treating pasted images like attached files with MIME metadata. Final submission converts image attachments to multimodal parts only when supported.

**Strengths:**
- Fastest path with least code churn.
- Likely improves most visible streaming artifacts immediately.
- Image paste can build on the existing attachment chip UX.

**Weakest link:** Addresses symptoms more than the rendering model, so edge-case glitching may remain.

**Risks:**
- May reduce but not fully eliminate streaming glitches caused by reparsing incomplete markdown every delta.
- Can leave the renderer conceptually fragile.

### V4. Adopt external TUI patterns wholesale

Use cursh/crush-inspired TUI patterns as the main design source: port their streaming collector, paste handling, and bottom-pane interaction model more broadly, potentially reshaping local chat/input/view composition to follow that design.

**Strengths:**
- Could materially improve UX consistency if the reference implementation is genuinely better.
- Opportunity to clean up multiple TUI pain points together.

**Weakest link:** The comparison target is not available, so this is design by vibes unless we locate the actual repo.

**Risks:**
- The referenced cursh repo is missing here, so the source of truth is incomplete.
- High scope creep risk: 'steal and rewrite' can turn into a broad redesign without solving the concrete bugs first.
