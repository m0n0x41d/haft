# Solutions for: Require baseline before measure in agent workflow

Problem: prob-20260330-005

## Variants (3)

### V1. Tool-layer baseline exposure + pre-measure blocker

Expose baseline in internal/tools/haft.go, add a store-backed check in the local measure path that blocks measurement when affected_files exist but no baseline hashes are stored, and update prompt/skill text to teach baseline before measure.

**Strengths:**
- Fixes the actual runtime path the agent uses.
- Preserves artifact-layer CL1 fallback for manual compatibility.
- Smallest change that prevents the agent from forgetting.

**Weakest link:** If another execution path bypasses the tool layer, weak measure without baseline remains possible.

**Risks:**
- Local and server behavior still differ slightly because artifact layer itself remains warning-based.

### V2. Artifact-layer hard enforcement

Change artifact.Measure itself to fail whenever affected_files exist but no baseline is present, forcing all callers—agent, CLI, server—to baseline first.

**Strengths:**
- Single source of truth; impossible to forget anywhere.
- Strongest FPF enforcement.

**Weakest link:** Too blunt if the project intentionally allows weak measurements in some contexts.

**Risks:**
- Potentially breaks existing workflows that relied on CL1 fallback.
- Higher compatibility risk.

### V3. Prompt-only and warning polish

Keep behavior unchanged but update prompts/docs and improve the warning text so the agent is more likely to do the right thing.

**Strengths:**
- Lowest risk.
- No behavior changes.

**Weakest link:** Advisory text is not enforcement.

**Risks:**
- Does not actually prevent the failure mode.
- Relies on the agent remembering, which is exactly what failed.
