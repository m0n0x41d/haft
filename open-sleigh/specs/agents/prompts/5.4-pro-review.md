## Thinking framework

Apply FPF (First Principles Framework) and semiotic engineering
discipline throughout. These resources are canonical:

### Core FPF distinctions to maintain throughout the review

- **Object ≠ Description ≠ Carrier.** The running engine (Object), these
  spec documents (Description), and the code-to-be (Carrier) are three
  different things. A type definition is a description; a passing test
  is evidence; a production run is reality. Don't collapse them.
- **Design-time ≠ Run-time.** A structural gate named `object_of_talk_is_specific` that only checks field presence is a design-time claim of meaning-enforcement; it does NOT mean the system prevents vague objects of talk at runtime. Flag places where a design-time mechanism is sold as a run-time guarantee.
- **Target system ≠ Enabling system.** The target system is Open-Sleigh (the harness engine). The enabling system is Elixir / OTP / the layered code + the team that writes it. Defects in enabling-system spec are code-quality issues. Defects in target-system spec are correctness issues — more severe.
- **Plan ≠ Reality.** A spec promises "observations never enter the Haft artifact graph." Does the proposed mechanism (module dependency graph) actually prevent this? Check the mechanism, not the promise.
- **Promise ≠ Delivery.** `ILLEGAL_STATES.md` promises 58 states will be enforced. Verify each proposed enforcement mechanism is actually sufficient to make the state unrepresentable. Distinguish "enforced by type" from "enforced by discipline."

### Semiotic checks

From `semiotics_slideument.md`, apply to every term, every type name,
every gate name, every prompt template reference:

1. **Object of talk.** Does each term refer to exactly one domain object?
2. **Context.** A term valid in one context may be wrong in another. "Phase" in Open-Sleigh (sum type) vs "phase" in a tracker workflow (their column names). Verify `TERM_MAP.md` disambiguates.
3. **Statement type.** Is this a description, a requirement, a promise, evidence, or a gate? Don't mix them.
4. **Lexicon vs ad-hoc.** Check every load-bearing noun against `TERM_MAP.md`. Any drift is a finding.
5. **Re-expression vs reinterpretation.** SPEC.md is a thin umbrella index; the normative content lives in `specs/`. Flag any place where SPEC.md carries authoritative normative prose instead of a pointer abstract (should have been migrated), or where a `specs/` document silently reinterprets what SPEC.md introduces.
6. **LADE decomposition (A.6.B).** Are boundary statements in the specs decomposed into Law / Admissibility / Deontics / Work-effect-Evidence, or do they conflate quadrants?

## Thai-disaster lens (load-bearing context)

Open-Sleigh's architecture is directly shaped by the pathology in
`~/Repos/hustles/thai/.context/audit-general-analysis.md`:

- 70K lines of tooling + 230K lines of governance docs vs ~1500 lines of actual product logic
- 99% of 28,560 commits from an AI agent, of which the bulk is the agent writing FOER / longforms / runbooks about its own process
- 105KB CLAUDE.md as recursive ceremony generator
- Three serial app rewrites, none in production

Open-Sleigh claims to structurally prevent recurrence via:

1. **L4 workspace-path allowlist:** agent cannot write into `open_sleigh/` or `~/.open-sleigh/`
2. **L1 bounded prose:** no narrative field exceeds 1000 chars
3. **L6 size budget:** `sleigh.md` > 300 lines fails compile
4. **L5 observation isolation:** `ObservationsBus` has zero compile-time path to `Haft.Client`

**Your review MUST test each of these claims.** For each:

- Does the proposed mechanism actually prevent the anti-pattern, or does it just slow it down?
- Is there a trivial bypass route? (e.g. agent writing to `/tmp/open-sleigh-fake/` then hard-linking; narrative bloat spread across multiple artifacts each < 1000 chars; `sleigh.md` under budget but prompts stored elsewhere; observations written via a proxy module that imports `Haft.Client`)
- What's the evidence the author would need to collect (post-`mix new`) to verify the claim?

## Review levels

### Level 1 — Target-system correctness (CRITICAL)

The specs must coherently model what Open-Sleigh **is**.

**Entity completeness:**

- Every entity in `TARGET_SYSTEM_MODEL.md` has a corresponding `Phase.t()` / `Session.t()` / etc. type declared?
- Every axis in `PHASE_ONTOLOGY.md` (Phase, GateKind, Verdict, AuthoringRole) has a corresponding entity field and a corresponding enforcement mechanism?
- Cross-reference `TERM_MAP.md` entries against both `TARGET_SYSTEM_MODEL.md` and `PHASE_ONTOLOGY.md` — every term referenced must have a canonical definition.

**Business-rule enforcement (in spec, to be verified in code later):**

- **PR5:** `Evidence.ref ≠ authoring artifact's self_id` — is this actually unrepresentable, or is the "authoring artifact's self_id" a value that a malicious / buggy caller could pass to `Evidence.new` legitimately? Verify that `Evidence.new` has access to the authoring artifact at construction.
- **PR10:** `PhaseOutcome` with `external_publication: true` requires `HumanGateApproval` — is the "separate constructor" design sound, or could a caller still pass `true` to the regular constructor? Check that the constructor path is genuinely bifurcated.
- **CL1–CL4 (v0.5 hybrid scoping):** Q-OS-4 resolved as hybrid — compile-time `@tool_registry` closed-atom function-clause dispatch per adapter + runtime `MapSet` per-phase scope. Verify the spec is honest: CL1 is claimed **type-level** (function-clause dispatch); CL2–CL3 are claimed **runtime-guard** (MapSet check). Confirm the labels match the mechanism described in `FUNCTIONAL_ARCHITECTURE.md L4` and `STACK_DECISION.md §Boundary enforcement`.
- **SE1:** Single-writer discipline — is there actually a mechanism preventing a second writer, or just a convention?

**Phase-graph completeness:**

- MVP-1 phase graph: Frame → Execute → Measure + terminal. Are all transitions explicit?
- MVP-2 phase graph: canonical names from Slide 12. Does `TARGET_SYSTEM_MODEL.md` declare `Workflow.mvp2/0` shape matching?

**Status-ontology soundness (4 axes):**

- Each axis has a declared sum type? Enumeration closed? Extension mechanism documented?
- Verdict on Measure cases (pass / fail / partial) — when does `:partial` ever fire in MVP-1 vs MVP-2? Flag if ambiguous.

**Illegal states coverage (58 total):**

### Level 4 — Thai-disaster anti-pattern check (CRITICAL)

For each of the 4 guardrails in `FUNCTIONAL_ARCHITECTURE.md §Thai-disaster architectural guardrails`:

**1. L4 workspace-path allowlist** (agent can't operate on Open-Sleigh's source)

- Is "cannot write into `open_sleigh/`" absolute, or can it be bypassed via symlink, absolute-path games, or `/tmp` scratch + install?
- Does the allowlist cover reads as well, or only writes?
- What happens if the agent runs `git clone` on Open-Sleigh's source into `workspace_path` and then edits it?

**2. L1 bounded prose** (`rationale ≤ 1000 chars`)

- Is there a way to smuggle prose via other fields? `evidence.kind`? `HumanGateApproval.reason` (bounded at 500)? Multiple artifacts each < 1000?
- Is the 1000 limit enforced at validation or silently truncated?

**3. L6 size budget** (`sleigh.md ≤ 300 lines`)

- Can prompt templates be inlined from other files? If `sleigh.md` references `@include ./long_prompt.md`, does the budget check follow includes?
- What about Markdown-embedded base64 or similar bloat?

**4. L5 observation isolation** (`ObservationsBus` no path to `Haft.Client`)

- Is `mix xref` actually precise enough to catch transitive dependencies?
- What if an intermediate module imports both?
- What if `ObservationsBus.emit/3` is called with a map containing a `Haft.ArtifactId.t()` value — does that indirect coupling count?

For each guardrail, rate: **strong / moderate / weak** — strong means
no bypass without architectural refactor; weak means convention is
doing the work.

## Meta-note

This is a pre-code review. The scale of "coverage" for illegal states
is necessarily at the mechanism-sufficiency level, not the test level
(because there are no tests yet). Be explicit about this distinction
in your findings — never claim a state is "enforced" when what you
mean is "the proposed mechanism, if implemented faithfully, would
enforce it."

The single most valuable thing you can do is name a bypass route for a
guardrail the author thought was airtight. The second most valuable is
flag a design-time claim that silently reduces to runtime discipline.
